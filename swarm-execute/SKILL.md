# Swarm Execute — Parallel Story Execution via Agent Teams

Execute `tasks/prd.json` stories in parallel using Claude Code's Agent Teams feature. Independent stories run concurrently in separate worktrees. Dependent stories batch after their prerequisites complete.

You are the **lead orchestrator**. You NEVER implement stories yourself — you spawn teammates, monitor progress, merge results, and enforce quality.

---

## When to Use

- `tasks/prd.json` exists (generated by `/shape-to-ralph` or `/plan-to-ralph`)
- At least 2 incomplete stories remain
- Stories have potential for parallel execution (independent file sets or explicit independence flags)
- Agent Teams is enabled (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`)

## When NOT to Use

- Only 1 incomplete story remains — just use Ralph (`ralph.sh`) or implement directly
- All stories are strictly sequential (each depends on the previous) — use Ralph instead
- No `tasks/prd.json` exists — run `/shape-to-ralph` or `/plan-to-ralph` first

## Fallback

If Agent Teams is not available or any prerequisite fails, tell the user:

```
Agent Teams not available. Fall back to sequential execution:
  .claude/skills/shape-to-ralph/scripts/ralph.sh
  — or —
  .claude/skills/plan-to-ralph/scripts/ralph.sh
```

## Step 1: Parse PRD

Read and parse context files:

1. **`tasks/prd.json`** — Extract all stories, their IDs, priorities, acceptance criteria, and notes
2. **`tasks/progress.txt`** — Understand what previous iterations accomplished
3. **`tasks/findings.md`** — Architecture decisions, known errors, patterns discovered

From each incomplete story's `notes` field, extract:

- **Files to CREATE** — New files this story introduces
- **Files to MODIFY** — Existing files this story changes
- **Affordance references** — N1, S2, U3 etc. that link to other stories
- **Source references** — "Slice VN from..." or "Task N from..." traceability

Store this parsed data — you'll need it for dependency analysis.

---

## Step 2: Dependency Analysis

Determine which stories can run in parallel. Three-pass analysis, conservative by default.

### Pass 1 — Text Scanning (strongest signal)

For each story, scan `notes` and `description` for dependency patterns:

```
Regex patterns to check:
  /depends on (V\d+|US-\d+)/i
  /requires .+ from (V\d+|US-\d+)/i
  /after (V\d+|US-\d+)/i
  /builds on (V\d+|US-\d+)/i
  /extends (V\d+|US-\d+)/i
  /uses .+ (created|built|defined) in (V\d+|US-\d+)/i
```

Also check `findings.md` slice dependency section — `/shape-to-ralph` already flags these:

```
"V5 is independent: could parallelize"
"V2 depends on V1: N2 wires to N3 (built in V1)"
```

### Pass 2 — File Overlap (hard signal)

Parse `CREATE:` and `MODIFY:` lines from each story's `notes`:

| Scenario                                                                      | Result                                                 |
| ----------------------------------------------------------------------------- | ------------------------------------------------------ |
| Story B `MODIFY`s a file story A `CREATE`s                                    | **B depends on A**                                     |
| Stories A and B both `MODIFY` the same file                                   | **Conflict — must serialize A before B** (by priority) |
| Stories A and B both `CREATE` different files                                 | **Independent**                                        |
| Story A `CREATE`s and story B `CREATE`s different files but in same directory | **Independent** (unless module index file)             |

**Special case — module index files**: If story A `CREATE`s `src/foo/foo.service.ts` and story B `MODIFY`s `src/foo/foo.module.ts` to register a new provider, B likely depends on A. Flag as dependency.

### Pass 3 — Affordance Cross-References (weak signal)

If story B's notes reference affordance IDs (N1, S2, U3) that are defined in story A's notes, B may depend on A. Use for confirmation only — do not create dependencies based solely on this pass.

### Conservative Default

**When uncertain, serialize.** False parallelization → merge conflicts + wasted tokens. False serialization → only costs time.

### Output

Build a dependency map:

```
Dependencies found:
  US-002 → depends on [US-001]    (file overlap: US-002 MODIFY payment.service.ts created by US-001)
  US-003 → depends on [US-001]    (text: "depends on V1")
  US-004 → depends on [US-002]    (file overlap: both MODIFY subscription.module.ts)
  US-005 → independent            (no file overlap, no text references)
```

---

## Step 3: Batch Generation

Group stories into parallel batches using topological sort on the dependency graph.

### Algorithm

1. **Batch 1**: All stories with zero dependencies (and not depending on any incomplete story)
2. **Batch N**: Stories whose ALL dependencies are in earlier batches
3. Within each batch, order by priority (lowest first) — this is the merge order

### File Conflict Detection

Within each batch, verify no two stories modify the same file:

- If conflict found: move the lower-priority story to the next batch
- Re-run conflict check until each batch is conflict-free

### Present Plan to User

```
Swarm Execution Plan
─────────────────────
Project: [name]
Branch:  [branch]
Stories: [N] incomplete of [M] total

Batch 1 (parallel):
  US-001 [P1]: [title]
  US-005 [P5]: [title]
  Files: no overlap ✓

Batch 2 (parallel, after batch 1):
  US-002 [P2]: [title]  ← depends on US-001
  US-003 [P3]: [title]  ← depends on US-001
  Files: no overlap ✓

Batch 3 (sequential, after batch 2):
  US-004 [P4]: [title]  ← depends on US-002
  Files: n/a (single story)

Estimated: 3 batches, max 2 concurrent teammates
Quality gates: [from prd.json qualityGates array]
Review gates: architect review + QA review per story (before merge)

Proceed? [Y/adjust/cancel]
```

Wait for user approval before continuing.

- **Y**: Proceed to team setup
- **adjust**: User modifies batches (e.g., "move US-003 to batch 3"), re-present
- **cancel**: Abort

## Step 4: Team Setup

### Create Agent Teams Tasks

For each incomplete story, create a task:

```
TaskCreate:
  subject: "[US-XXX] [story title]"
  description: [full story description + acceptance criteria + notes]
  activeForm: "Implementing [US-XXX] [story title]"
```

### Set Dependencies

For each story with dependencies:

```
TaskUpdate:
  taskId: [task ID for dependent story]
  addBlockedBy: [task IDs of dependency stories]
```

### Write Recovery State

Write `tasks/swarm-state.json`:

```json
{
  "teamName": "swarm-[branch]-[timestamp]",
  "prdFile": "tasks/prd.json",
  "batches": [
    {
      "batch": 1,
      "stories": ["US-001", "US-005"],
      "status": "pending"
    },
    {
      "batch": 2,
      "stories": ["US-002", "US-003"],
      "status": "pending"
    }
  ],
  "worktrees": {},
  "mergeOrder": [],
  "startedAt": "[ISO timestamp]",
  "completedAt": null
}
```

## Step 5: Batch Execution

Execute one batch at a time. Within each batch, spawn teammates in parallel.

### For Each Batch

1. Update `swarm-state.json`: set batch status to `"in_progress"`
2. For each story in the batch, spawn a teammate:

### Spawning a Teammate

Each teammate runs in its own worktree for isolation. Provide the teammate with context by filling the `teammate-prompt.md` template.

**Fill the template** (from `.claude/skills/swarm-execute/templates/teammate-prompt.md`):

| Placeholder               | Value                                                                                                            |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `{{STORY_ID}}`            | e.g., `US-001`                                                                                                   |
| `{{STORY_TITLE}}`         | From prd.json                                                                                                    |
| `{{STORY_DESCRIPTION}}`   | From prd.json                                                                                                    |
| `{{ACCEPTANCE_CRITERIA}}` | Formatted list from prd.json                                                                                     |
| `{{STORY_NOTES}}`         | Full notes field from prd.json                                                                                   |
| `{{QUALITY_GATES}}`       | Join `prd.json.qualityGates` array with `&&` (e.g., `pnpm typecheck && pnpm lint && pnpm test`)                  |
| `{{FINDINGS_CONTENT}}`    | Current content of `tasks/findings.md`                                                                           |
| `{{PROGRESS_CONTENT}}`    | Current content of `tasks/progress.txt`                                                                          |
| `{{PRD_DESCRIPTION}}`     | The `description` field from prd.json (contains source doc path)                                                 |
| `{{BRANCH_NAME}}`         | From prd.json                                                                                                    |
| `{{BATCH_SIBLINGS}}`      | Summary of other stories in this batch (ID, title, files touched) — gives teammates awareness of concurrent work |

**How to fill `{{BATCH_SIBLINGS}}`**: For each _other_ story in the same batch, include:

```
- US-XXX: [title] — Files: [CREATE/MODIFY list]. Modules: [affected modules].
```

If only one story in the batch, use: `"No other stories in this batch — you are the only teammate."`

**Spawn the teammate** using the Task tool with `isolation: "worktree"`:

```
Task:
  description: "[US-XXX] [short title]"
  prompt: [filled teammate-prompt.md content]
  subagent_type: "general-purpose"
  isolation: "worktree"
  run_in_background: true
```

Record the worktree info in `swarm-state.json`.

### Monitoring

After spawning all teammates in a batch:

1. Wait for all teammates to complete (use `TaskOutput` with `block: true` or poll `TaskList`)
2. If a teammate fails:
   - **Attempt 1-2**: Re-read the error, provide feedback, re-spawn with enhanced context
   - **Attempt 3**: Spawn a replacement teammate with the error history added to context
   - **Attempt 4+**: Mark story as blocked, ask the user for guidance

### Batch Completion

A batch is complete when all stories in the batch have their tasks marked as completed. Proceed to the review gate (Step 5.5) before merge.

## Step 5.5: Review Gate

After all stories in a batch complete their quality gates, run architect and QA reviews **per-story, before merge**. Reviews for different stories in the same batch run concurrently.

### For Each Completed Story (all stories' reviews run concurrently)

#### 1. Spawn Reviewers

Spawn an architect reviewer and a QA reviewer **in parallel** for each story. Both are read-only — they do NOT get a worktree.

**Fill the architect reviewer template** (from `.claude/skills/swarm-execute/templates/architect-reviewer-prompt.md`):

| Placeholder               | Value                                                |
| ------------------------- | ---------------------------------------------------- |
| `{{STORY_ID}}`            | e.g., `US-001`                                       |
| `{{STORY_TITLE}}`         | From prd.json                                        |
| `{{STORY_DESCRIPTION}}`   | From prd.json                                        |
| `{{ACCEPTANCE_CRITERIA}}` | Formatted list from prd.json                         |
| `{{STORY_NOTES}}`         | Full notes field from prd.json                       |
| `{{WORKTREE_BRANCH}}`     | The teammate's worktree branch name from swarm-state |
| `{{BASE_BRANCH}}`         | From prd.json `branchName`                           |

**Fill the QA reviewer template** (from `.claude/skills/swarm-execute/templates/qa-reviewer-prompt.md`) with the same placeholders.

**Spawn both reviewers**:

```
Task:
  description: "[REVIEW-ARCH] US-XXX"
  prompt: [filled architect-reviewer-prompt.md]
  subagent_type: "general-purpose"
  run_in_background: true

Task:
  description: "[REVIEW-QA] US-XXX"
  prompt: [filled qa-reviewer-prompt.md]
  subagent_type: "general-purpose"
  run_in_background: true
```

Note: Do NOT use `isolation: "worktree"` — reviewers are read-only and need access to the existing worktree branch via git.

#### 2. Wait for Both Reviewers

Use `TaskOutput` with `block: true` to wait for both reviewers to complete.

#### 3. Parse Results

Extract the JSON output from each reviewer's response. Look for the JSON block containing `verdict`, `blockers`, and `suggestions`.

#### 4. Determine Verdict

- **BOTH pass** (`verdict: "pass"`) → story proceeds to merge. Update swarm-state: `finalVerdict: "pass"`.
- **Either blocked** (`verdict: "blocked"`) → proceed to remediation (step 5 below).

#### 5. Remediation (if blocked)

Aggregate blockers from both reviewers and send a remediation prompt to the coding teammate.

**Fill the remediation template** (from `.claude/skills/swarm-execute/templates/remediation-prompt.md`):

| Placeholder               | Value                                                                  |
| ------------------------- | ---------------------------------------------------------------------- |
| `{{STORY_ID}}`            | Story identifier                                                       |
| `{{STORY_TITLE}}`         | Story title                                                            |
| `{{STORY_DESCRIPTION}}`   | Story description                                                      |
| `{{ACCEPTANCE_CRITERIA}}` | Acceptance criteria                                                    |
| `{{ARCHITECT_BLOCKERS}}`  | Formatted list of architect blockers (file, line, issue, fix)          |
| `{{QA_BLOCKERS}}`         | Formatted list of QA blockers (file, line, issue, testToWrite)         |
| `{{QUALITY_GATES}}`       | Join `prd.json.qualityGates` array with `&&` (same as teammate prompt) |

If one reviewer passed and the other blocked, only include the blocking reviewer's section. Set the other to `"No blockers from [reviewer type]."`.

**Send to the coding teammate** via `SendMessage`:

```
SendMessage:
  type: "message"
  recipient: [teammate name]
  content: [filled remediation-prompt.md]
  summary: "Review blockers found for US-XXX"
```

The teammate is idle in its worktree — the message wakes it up. Wait for the teammate to re-complete its task.

#### 6. Re-review (Attempt 2)

After the teammate fixes blockers and re-completes:

1. Re-spawn architect + QA reviewers (same process as step 1)
2. Parse results
3. **Pass** → proceed to merge. Update swarm-state: `finalVerdict: "pass"`, `attempts: 2`.
4. **Still blocked** → escalate to user

#### 7. Escalation (if still blocked after attempt 2)

Present the remaining blockers to the user and ask:

```
Review Gate — {{STORY_ID}} still blocked after 2 attempts

Remaining blockers:
  [list blockers with file:line references]

Options:
  1. Override — merge anyway (verdict: "overridden")
  2. Skip — skip this story, continue with remaining batch
  3. Abort — stop swarm execution
```

Update swarm-state with the user's choice.

### Update swarm-state.json

After all stories in the batch pass reviews, update `swarm-state.json` with the `reviews` object:

```json
{
  "reviews": {
    "US-001": {
      "architect": { "verdict": "pass", "blockerCount": 0, "suggestionCount": 2 },
      "qa": { "verdict": "pass", "blockerCount": 0, "suggestionCount": 1 },
      "attempts": 1,
      "finalVerdict": "pass"
    }
  }
}
```

Proceed to Step 6 (Merge) only after all stories in the batch have `finalVerdict: "pass"` or `"overridden"`.

## Step 6: Merge

After each batch passes the review gate (Step 5.5), merge teammate worktrees into the feature branch. Merge sequentially by priority (lowest first) even within parallel batches.

### Precondition

Every story in the batch must have `finalVerdict: "pass"` or `"overridden"` in `swarm-state.json` before merging. Stories with `finalVerdict: "skipped"` are excluded from merge.

### For Each Completed Story (by priority order)

1. **Verify review gate passed** — Check that the story's `finalVerdict` in `swarm-state.json` is `"pass"` or `"overridden"`
2. **Merge the worktree branch**:

```bash
git merge --no-ff [worktree-branch] -m "feat([scope]): [US-XXX] [story title]"
```

3. **Run quality gates post-merge** — execute every command from the `qualityGates` array in `prd.json`, joined with `&&`:

```bash
# Read from prd.json qualityGates array — do NOT hardcode commands
{{QUALITY_GATES}}
```

4. **If post-merge quality gates fail**:
   - Identify the conflict or regression
   - Fix it (minimal changes only — do NOT re-implement the story)
   - Commit the fix: `fix([scope]): resolve merge conflict from [US-XXX]`
   - Re-run quality gates

5. **Update `tasks/prd.json`**: Set `"passes": true` for the merged story

6. **Update `tasks/progress.txt`**: Append batch progress entry using the `swarm-progress.md` template

7. **Update `tasks/findings.md`**: Aggregate any new findings from the teammate's output (architecture decisions, errors, patterns)

8. **Update `swarm-state.json`**: Record merge, update batch status

### After All Stories in Batch Merged — Findings Aggregation

**This is the critical learning step.** Before moving to the next batch, aggregate ALL teammate findings into `tasks/findings.md`. The next batch's teammates will read this file at spawn time — their quality depends on what you write here.

1. **Collect teammate findings**: From each teammate's completion output, extract:
   - Architecture decisions and rationale
   - Errors encountered and resolutions (with specific error messages)
   - Codebase patterns discovered
   - Any warnings about potential merge issues with sibling work

2. **Collect review findings**: From each story's architect and QA reviews, extract:
   - Blockers that were fixed during remediation → add to "Errors Encountered" (so future teammates avoid the same mistakes)
   - Suggestions (not fixed) → add to "Review Suggestions" section with story ID, reviewer type, file:line, suggestion text
   - Patterns noted by reviewers → add to "Patterns Discovered"

3. **Update `tasks/findings.md`** sections:
   - **Architecture Decisions**: Append new decisions from this batch
   - **Errors Encountered**: Append all errors with attempt count and resolution (including review blockers that were fixed)
   - **Patterns Discovered**: Append new patterns (deduplicate with existing)
   - **Review Suggestions**: Append unfixed reviewer suggestions with story ID, reviewer type (architect/qa), file:line, suggestion text
   - **Resources**: Append any new file paths or references teammates found useful

4. **Update `tasks/progress.txt`**: Append batch progress entry using `swarm-progress.md` template

5. **Update `swarm-state.json`**: Set batch status to `"completed"`

6. **Proceed**:
   - If more batches remain → Step 5 for the next batch (teammates will read the updated findings)
   - If all batches complete → Step 7

**Why this matters**: In Ralph's sequential loop, each iteration naturally reads the previous iteration's findings. In swarm mode, this aggregation step is the only bridge between batches. Skip it and batch 2 teammates lose all of batch 1's learnings.

### Final Validation

Run quality gates one final time on the complete feature branch — execute every command from the `qualityGates` array in `prd.json`:

```bash
# Read from prd.json qualityGates array — do NOT hardcode commands
{{QUALITY_GATES}}
```

Then check for an optional `e2eCommand` field in `prd.json`. If present, run it (E2E tests only run at final validation, not per-story or per-batch):


- `tasks/prd.json`: Verify all stories show `passes: true`
- `tasks/progress.txt`: Append final summary entry

## Critical Rules

1. **NEVER implement stories yourself** — you are the orchestrator. All implementation happens in teammates.
2. **Each teammate gets its own worktree** — no two teammates share a working directory.
3. **Merges are sequential by priority** — even within a parallel batch, merge lowest priority first.
4. **Quality gates run per-teammate AND post-merge** — passing in isolation doesn't guarantee passing after merge.
5. **File conflicts prevent parallel execution** — if two stories modify the same file, they MUST be in different batches.
6. **Teammates do NOT touch shared files** — `prd.json`, `progress.txt`, `findings.md`, and `swarm-state.json` are lead-only. Teammates report via task completion and messages.
7. **Conservative dependency analysis** — when uncertain, serialize. Wasted time < wasted tokens from merge conflicts.
8. **User approves the plan** — never start spawning teammates without user confirmation of the batch plan.
9. **Recovery is possible** — `swarm-state.json` tracks progress. If the session dies, re-invoke `/swarm-execute` to resume from the last completed batch.
10. **Ralph is the fallback** — if Agent Teams fails or isn't available, tell the user to use `ralph.sh` instead.
11. **Reviews happen before merge** — no exceptions except user override. Every story must pass architect + QA review before its worktree branch is merged.
12. **Reviewers are read-only** — reviewer agents never modify files. They analyze the diff and produce structured JSON verdicts. Never spawn reviewers with `isolation: "worktree"`.
13. **Max 2 review attempts per story** — if a story is still blocked after the teammate's remediation and a second review, escalate to the user. Do not loop indefinitely.

---

## Recovery

If `/swarm-execute` is re-invoked and `tasks/swarm-state.json` exists:

1. Read `swarm-state.json`
2. Identify the last completed batch
3. Skip completed batches
4. Resume from the first incomplete batch
5. Re-run dependency analysis to confirm remaining batches are still valid (code may have changed)
6. Present updated plan to user and continue
